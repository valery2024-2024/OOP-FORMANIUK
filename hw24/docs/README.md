# Контрольні питання

## 1. Що таке рефакторинг і навіщо він потрібен?

**Рефакторинг** — це покращення внутрішньої структури коду без зміни роботи програми.

Його використовують для:

* покращення читабельності коду;
* зменшення складності;
* спрощення підтримки;
* легшого додавання нових функцій;
* зменшення кількості помилок.

Під час рефакторингу код стає чистішим і зрозумілішим для розробників.

## 2. Назвіть 5 найпоширеніших code smells та способи їх усунення

### 1. Long Method

**Проблема:** метод занадто довгий.

**Рішення:**

* Extract Method — розбити метод на менші.

### 2. Duplicate Code

**Проблема:** однаковий код повторюється.

**Рішення:**

* Extract Method — винести спільний код в окремий метод.

### 3. Magic Numbers

**Проблема:** у коді є числа без пояснення.

**Рішення:**

* Extract Constant — створити константи з назвами.

### 4. Large Class (God Object)

**Проблема:** клас має занадто багато відповідальностей.

**Рішення:**

* Extract Class — розділити клас на кілька менших.

### 5. Long Parameter List

**Проблема:** метод має забагато параметрів.

**Рішення:**

* Introduce Parameter Object — створити окремий клас для параметрів.

## 3. Що таке “God Object” і як його виправити?

**`God Object`** — це дуже великий клас, який виконує багато різних задач одночасно.

Наприклад:

* робота з базою даних;
* перевірка даних;
* обчислення;
* надсилання повідомлень;
* створення звітів.

Такий клас важко:

* читати;
* тестувати;
* змінювати.

Виправлення:

* розділити великий клас на декілька маленьких класів;
* кожен клас повинен мати одну відповідальність (SRP).

## 4. Чому важливо мати тести перед рефакторингом?

Тести потрібні для перевірки того, що рефакторинг не зламав програму.

Без тестів можна:

* випадково змінити логіку;
* отримати неправильні результати;
* не помітити помилку.

Тести дозволяють безпечно змінювати структуру коду.

## 5. Що таке цикломатична складність і як вона впливає на якість коду?

Цикломатична складність — це показник складності коду.

Вона залежить від:

* кількості if;
* switch;
* циклів;
* умов.

Чим більша складність:

* тим важче тестувати код;
* тим більше шансів на помилки;
* тим важче підтримувати програму.

Низька складність означає, що код простіший і зрозуміліший.
